Enterprise vCenter MCP

v2 Implementation Plan

Plan v1  ·  Feb 2026

Context

Transform Enterprise_vCenter_MCP/ from a copy of the v1 Mac/Claude Desktop MCP server into a full enterprise system. v1 required Claude Desktop + SSH tunnel on the user's Mac. v2 is always-on, browser-accessible, hosted on OCI Compute (ap-melbourne-1), using OCI GenAI (Cohere Command A) as the LLM, LangChain as the agent/MCP client, and OCI PostgreSQL + pgvector for RAG over runbooks.

Architecture

Browser (team) → :8501 ↓ [app container] Streamlit UI + LangChain ReAct Agent │ HTTP/SSE :8080 (internal Docker network) [mcp_server container] FastMCP + pyVmomi │ TCP 443 Customer vCenter (pre-established connectivity) [app container] also calls: → OCI GenAI API (Cohere Command A + Cohere Embed) via Instance Principal → OCI PostgreSQL (pgvector) managed DB, always-on ↑ runbook PDFs/MDs ingested once on deploy

Key Decisions

Compute
OCI Compute VM + Docker Compose
Always-on, SSH access, no Container Instance complexity. VM.Standard.E4.Flex 2 OCPU / 8 GB.
Vector Store
OCI PostgreSQL + pgvector
Managed DB — no volume management, no data loss on restart, enterprise-grade persistence. Replaces ChromaDB.
LLM + Embeddings
OCI GenAI Cohere Command A + Embed v3
Stays entirely within OCI tenancy. Matches reference RAG project patterns. Melbourne region.
MCP Transport
HTTP / SSE
FastMCP already supports it. mcp_server runs as a proper Docker service on :8080, not a subprocess.
MCP Client
langchain-mcp-adapters
Official LangChain package. MultiServerMCPClient wraps MCP tools as LangChain tools for the agent.
OCI Auth
Instance Principal
No API keys on VM. Docker containers on OCI VM inherit IMDS token. api_key fallback for local dev.
vCenter Connectivity
Customer-managed
Customer pre-establishes network (FastConnect / VPN / OCVS peering). App just takes host + credentials.
Region
ap-melbourne-1
vCenter is there. OCI GenAI available. Low latency. All services co-located.

Key Code Patterns

OCI GenAI LLM (from reference rag_chain.py)

from langchain_community.chat_models.oci_generative_ai import ChatOCIGenAI

llm = ChatOCIGenAI(
    auth_type="INSTANCE_PRINCIPAL",   # uppercase — required
    model_id="cohere.command-a-03-2025",
    compartment_id=COMPARTMENT_ID,
    service_endpoint=OCI_GENAI_ENDPOINT,
    model_kwargs={"temperature": 0, "max_tokens": 2048},
)

LangChain ↔ MCP SSE (langchain-mcp-adapters)

from langchain_mcp_adapters.client import MultiServerMCPClient

async def _get_mcp_tools():
    client = MultiServerMCPClient({
        "vcenter": {"transport": "sse", "url": "http://mcp_server:8080/sse"}
    })
    return await client.get_tools()   # returns LangChain tools list

Async bridge (Streamlit + asyncio)

import nest_asyncio
nest_asyncio.apply()   # patches Tornado loop — call once at module top

def get_mcp_tools():
    return asyncio.run(_get_mcp_tools())   # sync wrapper for Streamlit

PGVector (OCI PostgreSQL)

from langchain_postgres import PGVector

vectorstore = PGVector(
    embeddings=build_embeddings(),
    collection_name="vcenter_runbooks",
    connection=PG_CONNECTION_STRING,   # postgresql+psycopg://user:pass@host:5432/db
)

from langgraph.prebuilt import create_react_agent

agent = create_react_agent(

Gotchas to Watch

AreaIssueFix
MCP server startup VCENTER_HOST read at module import — crashes if missing All vCenter vars in compose env before container starts
FastMCP SSE URL Endpoint is /sse not /mcp Use http://mcp_server:8080/sse in MultiServerMCPClient
Streamlit + asyncio RuntimeError: Event loop already running nest_asyncio.apply() at top of agent.py before any async code
Vector store persistence Lost on container restart if file-based (ChromaDB) OCI PostgreSQL (managed) — data survives restarts, reboots, redeploys
Instance Principal in Docker Does IMDS work from containers on OCI VM? Yes — OCI IMDS (169.254.169.254) accessible from Docker containers
auth_type casing LangChain OCI needs uppercase string "INSTANCE_PRINCIPAL" not "instance_principal"
App starts before MCP ready get_mcp_tools() fails on startup depends_on: condition: service_healthy + healthcheck on mcp_server
OCI firewalld Security list rules alone not enough on Oracle Linux 8 firewall-cmd --add-port=8501/tcp on VM after provisioning
psycopg version langchain-postgres uses psycopg v3, not psycopg2 pip install psycopg[binary] (v3), not psycopg2
Python version pyVmomi has wheel issues on Python 3.13 slim Use python:3.12-slim in both Dockerfiles

Verification Steps

  • docker compose build — both images build without error
  • docker compose up — mcp_server passes healthcheck, app starts cleanly
  • Open http://localhost:8501 — Streamlit UI loads with Oracle logo in sidebar
  • Ask: "How many VMs are powered on?" — agent calls get_inventory_summary via MCP → live vCenter data
  • Drop a runbook PDF into runbooks/, run ./scripts/ingest_docs.sh
  • Ask: "What's the DR procedure for cluster X?" — agent calls search_runbooks + vCenter tools, synthesises answer
  • On OCI VM: verify Instance Principal works — GenAI call succeeds without any API key
